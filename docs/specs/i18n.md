# Thai And English Localization

## Task

Add production-quality Thai and English localization to the entire application.
Follow `AGENTS.md`, preserve existing routes, behavior, test IDs, mock APIs, and
feature tests. Use English as the fallback locale.

## Architecture

- Use `next-intl` with the Next.js App Router. Do not build a custom translation
  system.
- Keep existing URLs such as `/login` and `/inbox`; do not add locale prefixes.
- Store translations in `messages/en.json` and `messages/th.json` with identical
  nested keys grouped by common UI and feature.
- Resolve locale from the saved user/workspace preference or locale cookie, then
  fall back to English.
- Add the provider at the correct shared layout boundary without turning the
  whole application into a Client Component.
- Add a reusable language switcher to the authenticated header and Settings.
  Changing language must update immediately and persist across refresh/login.

## Translation Rules

- Translate navigation, headings, labels, buttons, filters, validation,
  confirmations, feedback, empty/loading/error states, accessibility labels,
  and user-facing metadata.
- Do not translate customer names, provider names, order numbers, message
  content, email addresses, SKUs, or API data unless localized data exists.
- Replace hard-coded user-facing strings; do not translate by DOM manipulation.
- Keep translation keys stable and never use an English sentence as a key.
- Map stable API error codes to translated messages on the client. Do not make
  the API return locale-dependent error strings.
- Format dates, times, numbers, percentages, and THB currency with the selected
  locale and workspace timezone.
- Thai text must wrap naturally without clipping, overlap, or forced
  character-by-character breaks.

## Mock API

Persist `en` or `th` through the existing Settings language preference. Reject
unsupported locale values. Do not add a database in this task.

## Playwright Tests

Create localization E2E coverage for:

- English is used as the fallback locale
- Switching to Thai updates visible UI without a full logout
- Selected language persists after refresh and navigation
- Login, Inbox, Orders, Dashboard, Publishing, Integrations, Customers, Team,
  and Settings render important text in both languages
- Date, time, number, percentage, and THB formatting follow the active locale
- Unsupported locale values fall back safely
- Thai desktop and mobile layouts do not clip, overlap, or horizontally overflow
- Existing test IDs and all existing feature tests continue to pass

Run lint, TypeScript checks, all Playwright tests, and production build. If any
check fails, fix it and rerun everything until all checks pass. Review the HTML
report before finishing.

## Acceptance Criteria

- Every user-facing application screen supports Thai and English.
- Language switching is immediate, persistent, accessible, and type-safe.
- Existing routes, APIs, permissions, layouts, and workflows have no regression.
- Translation files have matching keys and no visible missing-key placeholders.
- All checks and Playwright tests pass.
